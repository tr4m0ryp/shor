<sast_methodology>
**SECURE CODE REVIEW METHODOLOGY (anchored in OWASP Code Review Guide v2 and CodeQL/Semgrep taint conventions):**

You MUST treat this analysis as taint-driven dataflow tracing, NOT pattern-matching keyword grep.

**1. Sources (untrusted data origins).** Identify every entry point where attacker-controlled data enters the system:
   - HTTP request fields: query string, path params, headers (incl. `Host`, `Referer`, `X-Forwarded-*`, `User-Agent`, `Cookie`), body (JSON, form, multipart), uploaded file content/name/type.
   - WebSocket / SSE / gRPC / GraphQL inputs.
   - Message queue payloads (Kafka, RabbitMQ, SQS, Redis pub/sub) that originate from network-reachable producers.
   - Webhooks, OAuth callbacks (incl. `state`, `code`, `id_token` claims), SAML assertions.
   - External API responses fed back into the application (treat as semi-trusted).
   - Database rows whose contents were originally user-supplied (second-order injection / stored XSS).

**2. Propagators.** Trace how tainted data moves through assignments, function calls, ORM hydration, template rendering, serializers, and async boundaries. Note explicit type casts, JSON parsing, and reflection — these can carry or strip taint depending on the language. Reference: Semgrep taint mode propagators (https://semgrep.dev/docs/writing-rules/data-flow/taint-mode/).

**3. Sanitizers.** Identify each function that legitimately neutralizes taint for a given context. A sanitizer for one sink class is often useless for another:
   - HTML-encode neutralizes HTML body context but NOT JS string context, URL context, or CSS context.
   - SQL parameterization neutralizes SQL but NOT command injection.
   - `path.Clean` / `filepath.Clean` does NOT prevent directory traversal alone — it must be combined with chroot/base-prefix validation.
   - Allowlist validators are stronger than blocklists; record which is in use.

**4. Sinks.** Enumerate dangerous sinks per CWE category (see language matrix below). For each sink, record: file:line, sink function, the variable carrying tainted data, the sanitizer (if any), and whether the path from source to sink is reachable from a network-exposed entry point.

**5. Trust boundaries.** Mark every crossing where data transitions between privilege zones (browser → reverse proxy → app → database, tenant A → tenant B, unauthenticated → authenticated, user → admin). Authorization checks SHOULD live at every boundary; missing checks are CWE-862 / CWE-863 findings.

# Penetration Test Scope & Boundaries

**Primary Directive:** Your analysis is strictly limited to the **network-accessible attack surface** of the application. All subsequent tasks MUST adhere to this scope. Before reporting any finding (entry point, sink, vulnerability), you MUST first verify it meets the "In-Scope" criteria below.

### In-Scope: Network-Reachable Components
A component is **in-scope** if its execution can be initiated, directly or indirectly, by a network request the deployed application server can receive. This includes:
- Publicly exposed web pages and API endpoints.
- Endpoints requiring authentication via the application's standard login mechanism.
- Any developer utility, debug console, admin panel, or script mistakenly exposed through a route or callable from other in-scope, network-reachable code.

### Out-of-Scope: Locally Executable Only
A component is **out-of-scope** if it CANNOT be invoked through the running application's network interface and requires an execution context external to the request-response cycle. This includes:
- Command-line interfaces (`go run ./cmd/...`, `python scripts/...`, `rails console`).
- IDE-internal tooling (run buttons, REPLs).
- CI/CD pipeline scripts and build tools (Dagger, Bazel, Make targets).
- Database migration scripts, backup tools, maintenance utilities.
- Local development servers, test harnesses, debug-only utilities.
- Static files or HTML pages that require manual browser opening (not served by the app).

## 9. XSS Sinks and Render Contexts
**TASK AGENT COORDINATION:** Populate from the XSS/Injection Sink Hunter Agent.

**Network Surface Focus:** Only report XSS sinks on web app pages or publicly facing components. EXCLUDE sinks in local-only scripts, build tools, developer utilities, or files that require manual browser opening.

You MUST include enough information to locate the exact line (file:line) for downstream agents. Use the finding template per sink.

**Definitions (per OWASP DOM XSS Prevention Cheat Sheet):**
- **XSS Sink:** A function or property that renders user-controllable data on a page.
- **Render Context:** Where the data lands within the page — determines required encoding.

Cover these contexts (map all to CWE-79 unless otherwise noted):

### HTML Body Context
- `element.innerHTML`, `element.outerHTML`
- `document.write()`, `document.writeln()`
- `element.insertAdjacentHTML()`
- `Range.createContextualFragment()`
- React `dangerouslySetInnerHTML`, Vue `v-html`, Angular `[innerHTML]` w/o `DomSanitizer`, Svelte `{@html}`
- jQuery sinks: `add()`, `after()`, `append()`, `before()`, `html()`, `prepend()`, `replaceWith()`, `wrap()`
- Server-side template `\|safe` / `mark_safe` / `raw` / `html_safe` / `@Html.Raw`

### HTML Attribute Context
- Event handlers set dynamically: `onclick`, `onerror`, `onmouseover`, `onload`, `onfocus`, etc.
- URL-bearing attributes: `href`, `src`, `formaction`, `action`, `background`, `data`, `poster`
- `style` attribute (CSS injection, CWE-79/CWE-94)
- Iframe `srcdoc`
- Generic attributes when quotes are escapable: `value`, `id`, `class`, `name`, `alt`

### JavaScript Context
- `eval()`
- `Function()` constructor
- `setTimeout()` / `setInterval()` with string argument
- Inline `<script>` containing server-templated data without context-aware escaping

### CSS Context
- `element.style.*` properties (especially `backgroundImage`, `content`)
- Inline `<style>` with user data
- `style` attribute with user data

### URL Context (open redirect / XSS via `javascript:` scheme — CWE-601 + CWE-79)
- `location`, `location.href`
- `location.replace()`, `location.assign()`
- `window.open()`
- `history.pushState()`, `history.replaceState()`
- `URL.createObjectURL()`
- jQuery selector with user input: `$(userInput)` (older versions, CVE-2020-11023)

### Server-Side XSS Sinks (often missed by JS-only reviews)
- Email templates rendered with user data (stored XSS via email then reflected in admin UI)
- PDF generators reflecting user input into HTML-to-PDF pipelines (PDF XSS, SSRF chain)
- Markdown rendering without sanitization (`marked`, `markdown-it`) + raw HTML pass-through
- JSON responses served with `Content-Type: text/html` (CWE-79 + CWE-430)
