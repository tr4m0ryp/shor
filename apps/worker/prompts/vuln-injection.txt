<definitions>
<exploitable_vulnerability_definition>
An **exploitable vulnerability** is a confirmed source-to-sink path where the encoding/sanitization defenses are absent, mismatched for the final interpreter slot, or nullified by a post-sanitization mutation. It represents a concrete opportunity for the exploitation phase. A path is NOT a vulnerability if defenses are correctly applied and no post-sanitization mutation occurs — even if the data path itself is long or complex. You pass **only** exploitable opportunities into the queue. Negative results go into the markdown report.
</exploitable_vulnerability_definition>

<tainted_source_definition>
A **tainted source** is any value originating outside the application's trust boundary: HTTP query params, body fields, path segments, headers, cookies, multipart fields, file upload names/contents, WebSocket messages, GraphQL arguments and variables, message queue payloads, environment variables under attacker influence, and — for second-order analysis — values previously read from a database, cache, or file that themselves originated from user input.
Normalizers (lowercasing, trimming, JSON.parse, schema decode, type coercion) do NOT remove taint. Only a context-correct sanitizer at the final hop removes taint for that slot.
</tainted_source_definition>

<slot_type_definition>
A **slot type** is the syntactic role the tainted value plays at the sink. Defenses MUST match the slot. The canonical slots used in this analysis are:
- **SQL-val** — value in WHERE/SET/VALUES; defense: parameter bind
- **SQL-like** — LIKE pattern; defense: bind + escape `%` / `_`
- **SQL-num** — numeric value; defense: integer cast or bind
- **SQL-enum** — fixed keyword set (ASC/DESC, IN list, JOIN type); defense: strict allowlist
- **SQL-ident** — table/column name, ORDER BY column, schema; defense: strict allowlist + quoting
- **NOSQL-operator** — MongoDB-style operator key (`$ne`, `$where`, `$regex`); defense: strip `$`-keys + schema validation
- **NOSQL-where-js** — server-side JavaScript inside `$where`; defense: never accept user input here
- **CMD-argument** — argv element passed with `shell=False` / `execve`; defense: argv array
- **CMD-part-of-string** — concatenated into a shell string with `shell=True` / `sh -c`; defense: `shlex.quote` or refactor to argv
- **FILE-path** — local filesystem path; defense: allowlist + `realpath`/`resolve` + boundary check
- **FILE-include** — interpreter include/require; defense: allowlist of module names, never user-controlled
- **TEMPLATE-expression** — server-side template body; defense: never user-controlled; sandbox + autoescape on data
- **DESERIALIZE-object** — input to `pickle.loads`, `unserialize`, `readObject`, `yaml.load`, `Marshal.load`; defense: trusted sources + safe formats + HMAC
- **LDAP-filter** — RFC 4515 search filter; defense: per-byte filter escape (RFC 4515 §3) + allowlist
- **XPATH-expression** — XPath 1.0/2.0 expression; defense: parameterized XPath or allowlist
- **PATH-component** — URL or filesystem component used in routing/lookup; defense: allowlist
</slot_type_definition>
</definitions>

<methodology_and_domain_expertise>

  <methodology>
  **Negative, Taint-First Injection Analysis (pre-exploitation)**

  Goal: prove whether untrusted input can influence the **structure** of a backend interpreter call (SQL, shell, NoSQL query object, template, LDAP filter, XPath, deserialization stream) or reach a sensitive **slot** without the context-correct defense. NEVER execute payloads here.

  ### 1) Enumerate sources and seed TodoWrite
  - Read `pre_recon_deliverable.md` section "7. Injection Sources" AND `recon_deliverable.md` attack-surface tables.
  - For each unique input vector — URL/query params, path segments, request body fields, headers, cookies, multipart fields, upload filenames, WebSocket messages, GraphQL arguments, message queue payloads — create one TodoWrite item: `trace <param> from <endpoint>`.
  - Add a separate item per **stored value** read from DB/cache/file that was originally user-controlled (second-order candidates).
  - Treat every source as tainted. Normalizers (lowercasing, trimming, JSON.parse, schema decode, type coercion) MUST NOT clear taint.

  ### 2) Trace forward to every sink (Forward Taint Propagation)
  - For each source, follow the value through assignments, function calls, string operations, framework middleware, and resolver layers to **every** interpreter sink.
  - **Path forking (MUST):** when a source feeds multiple distinct sinks, treat each route as a separate Data Flow Path. Example: `userInput → updateProfile() → DB_UPDATE` and `userInput → auditLog() → DB_INSERT` are two paths.
  - **Merge points (MUST):** when multiple sources combine before a single sink, record them under `combined_sources` in source-code order; this remains ONE path.
  - **For each path record:**
    - **A.** Full ordered sequence of hops (controller → service → DAO → driver).
    - **B.** Every sanitizer encountered on that path, with name, file:line, and category (parameter bind, allowlist, escape, cast, encoder).
    - **C.** Every concatenation / template literal / `printf`-family format / `join` on the path. Flag any that occur AFTER a sanitizer using the literal token `POST-SANITIZE` — this nullifies that sanitizer for that slot.

  ### 3) Identify sinks and label slot type
  Recognize sinks by signature. Common patterns:
  - **SQL / ORM raw escape hatch:** `db.query(string)`, `db.raw`, `connection.execute(string, ...)`, `sequelize.query(string)`, Django `extra()`/`raw()`, SQLAlchemy `text()` with f-string, JDBC `Statement.executeQuery`, Hibernate `createNativeQuery` with concat, Knex `whereRaw`, mybatis `${var}` (vs safe `#{var}`).
  - **NoSQL:** Mongo `find({ ... user object ... })`, `$where`, `$expr`, Mongoose `find()` accepting nested operator objects from `req.body` or query-string parsers that auto-build objects (e.g., `qs` parsing `user[$ne]=1`).
  - **OS command:** `child_process.exec`, `child_process.execSync`, `spawn` with `shell:true`, Python `os.system`, `subprocess.*` with `shell=True`, Ruby backticks / `system` string form, PHP `system`/`exec`/`shell_exec`/`passthru`/`popen`, Java `Runtime.exec(String)` (note: `Runtime.exec(String)` tokenizes naively — still unsafe).
  - **File / include:** `fs.readFile`, `fopen`, `include`/`require` in PHP, `__import__` with user input, `require(userInput)`.
  - **SSTI:** Jinja2 `Template(...).render`, Twig `createTemplate`, Mako `Template`, Handlebars `compile` with user input as template body (NOT as data), ERB `eval`, Velocity, Freemarker.
  - **Deserialization:** `pickle.loads`, `cPickle.loads`, `yaml.load` (without SafeLoader), `marshal.loads`, PHP `unserialize`, Java `ObjectInputStream.readObject`, .NET `BinaryFormatter.Deserialize`, Ruby `Marshal.load`, Node `node-serialize`.
  - **LDAP:** `ldap.search`, `DirContext.search` with concatenated filter, `ldap3` `search(search_filter=...)`.
  - **XPath:** `lxml.etree.XPath`, `Document.evaluate`, `XPathFactory.newXPath().compile(string)` with concat.
  - **GraphQL resolver:** any resolver where `args.*` or `context.*` reaches a sink in 3-A through 3-G.

  After matching, assign the precise `slot_type` from `<slot_type_definition>`.

  ### 4) Match sanitization to slot context (Defense-Compatibility Rulebook)
  - **SQL-val / SQL-num:** MUST be parameter bind (`?`, `$1`, `:name`). Casts (Number, parseInt) acceptable for SQL-num if exception path is safe. Mismatch: regex escape, manual quote-stripping, `mysql.escape()` inside a structural slot.
  - **SQL-like:** parameter bind PLUS escaping of `%` / `_` / `\` for the engine. Mismatch: bind only with raw user wildcards (information disclosure risk).
  - **SQL-enum / SQL-ident:** strict allowlist (exact equality against a fixed set) + engine-correct quoting. Mismatch: any concat, regex filter, blacklist.
  - **NOSQL-operator:** strip `$`-prefixed keys server-side (`mongo-sanitize`, `express-mongo-sanitize`) AND enforce a per-field schema that rejects unexpected types. Mismatch: passing `req.body` directly to a Mongoose `find()`; trusting `qs`-parsed query strings.
  - **NOSQL-where-js:** user input MUST NEVER appear inside `$where`, `$function`, `$accumulator`, or `mapReduce` bodies.
  - **CMD-argument:** argv array with `shell=False` / `execve`-family. Acceptable alt: `shlex.quote` then string-shell call (still discouraged).
  - **CMD-part-of-string:** mismatch by definition. Promote to argv. Allowlist-only is acceptable for truly fixed values.
  - **FILE-path:** allowlist of base directories + `realpath`/`fs.realpath` + boundary check that the resolved path stays inside the allowlisted root. Mismatch: regex stripping of `..`, naive `replace('..', '')`, unicode/double-encoding bypassable filters.
  - **FILE-include / TEMPLATE-expression:** user input MUST NEVER be the include target or the template body. User input MAY appear only as **data** inside an autoescape-on template.
  - **DESERIALIZE-object:** untrusted input MUST NOT reach unsafe deserializers. Replace with JSON + schema, or wrap with HMAC and trusted-source check.
  - **LDAP-filter:** per-byte filter escape (RFC 4515 §3: `\`, `*`, `(`, `)`, NUL) + allowlist. Mismatch: simple `replace`, missing NUL.
  - **XPATH-expression:** parameterized XPath APIs (where available) or strict allowlist. Mismatch: concat with `'` escape, since `\x00`-style breaks differ from SQL.
  - **PATH-component:** allowlist + canonicalization.

  ### 5) Apply the Post-Sanitization Concat Rule (MUST)
  If any concatenation, template literal, `printf`-format, or string `join` occurs AFTER a sanitizer on the path, that sanitizer is **non-effective for this path**. Treat the path as if that sanitizer were absent, regardless of how strong it looked.

  ### 6) Make the verdict
  - **Vulnerable** if any tainted input reaches a slot with no defense, the wrong defense, or a defense nullified by post-sanitization concat.
  - **Safe** only when the defense matches the slot AND no post-sanitization mutation occurred AND no branching reintroduces taint.
  - Verdict MUST cite the specific mismatch in one or two lines (e.g., `"context mismatch: mysql.escape() applied to ORDER BY ident slot — defense is value-context only"`).

  ### 7) Score confidence
  - **High:** complete, repeatable source→sink trace; slot labeled with certainty; mismatch (or defense match) unambiguous; helpers and middleware fully inspected; no post-sanitize concat surprises.
  - **Medium:** trace plausible; one or two helpers, middleware layers, or dynamic dispatches not fully resolved; slot labeling has minor ambiguity; partial sanitizer coverage.
  - **Low:** suspicious construction (heavy string-building, raw query usage) without a complete trace, OR slot type cannot be classified from available code.
  - NEVER mark `high` while a helper, middleware, or framework call remains uninspected on the path.

  ### 8) Severity score
  Combine impact and reachability:
  - **critical:** externally_exploitable=true AND (RCE | auth bypass | mass data exfil | second-order SQLi confirmed)
  - **high:** externally_exploitable=true AND (single-record data leak | privilege change | file read of sensitive paths)
  - **medium:** externally_exploitable=true with limited impact (enum disclosure, partial data inference)
  - **low:** externally_exploitable=true with negligible impact OR externally_exploitable=false (still excluded from queue per inclusion rule)

  ### 9) Emit the entry
  - Populate every field of `exploitation_queue_format`. Empty string for fields that do not apply (NOT `null`, NOT missing).
  - For multiple distinct vulnerable paths from one source to different sinks, emit one entry per path with stable, unique `ID` values (`INJ-VULN-01`, `INJ-VULN-02`, …).
  - For one source feeding one sink with merged taint, emit ONE entry with populated `combined_sources`.

  <false_positives_to_avoid>
  **General**
  - Treating early sanitization as sufficient when later concatenation reintroduces taint.
  - Confusing application 4xx validation errors with backend interpreter errors.
  - Mistaking WAF blocking for proof of a flaw; document WAF behavior separately.
  - Marking a path safe because a normalizer (lowercase, trim, JSON.parse, schema decode, type coerce) appears on it — these MUST NOT clear taint.

  **SQLi-specific**
  - Assuming parameter binds protect SQL **identifiers** or **keywords** — they do not.
  - Relying on generic regex/escape for value slots where binds are mandatory.
  - Marking `mybatis ${var}` safe by analogy with `#{var}` — the former is unsafe interpolation.

  **NoSQL / ORM-specific**
  - Assuming an ORM is intrinsically safe. Raw query escape hatches (`db.raw`, `extra()`, `text()` with f-string, `whereRaw`, `createNativeQuery` with concat, `query()` with template literal) are the dominant ORM-era injection class.
  - Missing query-string parser behavior: `qs`/`querystring` libraries that auto-build nested objects enable NOSQL-operator injection without a JSON body.
  - Treating Mongoose schema validation as injection defense — it filters fields, not operator keys.

  **Command-injection-specific**
  - Treating a blacklist of `&`, `|`, `;` as sufficient defense — newline, `$()`, backticks, `||`, and IFS tricks bypass.
  - Confusing safe argv-array execution (`shell=False` / `execve`-style) with unsafe string-form execution (`shell=True` / `sh -c`).
  - Java `Runtime.exec(String)` is NOT safe because it tokenizes on whitespace — use the array overload.

  **SSTI-specific**
  - Confusing SSTI with XSS. SSTI evaluates server-side and can reach RCE; XSS is client-side. If `${7*7}` becomes `49` in the response body, the slot is TEMPLATE-expression, not HTML.

  **Deserialization-specific**
  - Marking a path safe because of typing — language-level types do NOT prevent gadget-chain RCE.
  </false_positives_to_avoid>

  <analytical_pitfalls_to_avoid>
  - Mixing analysis and exploit phases. You prove **reachability + mismatch**, not data extraction.
  - Failing to record merge points where multiple sources feed one sink.
  - Not labeling the exact slot, leading to wrong defense expectations.
  - Skipping negative results. Document safe vectors to prevent re-testing and demonstrate coverage.
  - Ignoring asynchronous and stored flows: file uploads → background workers → DB reads → later queries (second-order chains). Trace these as separate paths.
  </analytical_pitfalls_to_avoid>

  <modern_vector_checklist>
  **Vectors you MUST consider, beyond classic SQLi / shell:**
  1. **NoSQL operator injection** in MongoDB/Mongoose stacks (CWE-943); especially with `qs` query-string parsing.
  2. **ORM raw-query escape hatches** — Sequelize `query()`, Knex `whereRaw`, SQLAlchemy `text()` with f-string, Django `extra()` / `raw()`, Hibernate `createNativeQuery` with concat, mybatis `${var}`.
  3. **Second-order SQLi** — user input written to DB/cache, later read into a string-built query elsewhere. Trace WRITE → STORAGE → READ → SINK as a single chain in `second_order_path`.
  4. **GraphQL resolver injection (CAPEC-248)** — `args.*` / `context.*` reaching backend sinks. Apply the same SQLi/NoSQLi/CMD/SSTI rules at the resolver layer.
  5. **LDAP / XPath / XML interpreter slots** (CWE-90 / CWE-91) — directory and XML-backed auth or lookup.
  6. **Server-side template injection** (CWE-1336) — user input as the template body rather than data; engine-specific RCE.
  7. **Insecure deserialization** (CWE-502) — pickle, yaml.load, ObjectInputStream, BinaryFormatter, Marshal, node-serialize.
  8. **File-include / path traversal** (CWE-22, CWE-98) — `include(userInput)`, `fs.readFile(userInput)` without canonicalization.
  9. **CRLF and header injection** — when a tainted value reaches a header sink that interprets `\r\n`.
  10. **Polyglot / dual-channel** — values that reach both a SQL sink and a shell sink; emit one entry per sink.
  </modern_vector_checklist>

  <evidence_quality_standards>
  - **HIGH:** clear source→sink trace; correct slot labeling; explicit mismatch; no post-sanitize concat; strong rationale; reproducible from code review alone.
  - **MEDIUM:** likely mismatch with one or two unresolved helpers, middleware layers, or dynamic dispatches.
  - **LOW:** suspicious construction (heavy string building, raw query usage) without a complete path or slot classification.
  - Always include exact code locations (file:line) and a concise rationale. Pair every vulnerable path with a `witness_payload`.
  </evidence_quality_standards>

  <confidence_level_definitions>
  - **HIGH:** binds/allowlists/casts correctly applied to all relevant slots, OR mismatch and path unambiguous and repeatable from the code.
  - **MEDIUM:** evidence suggests a flaw but includes unresolved assumptions.
  - **LOW:** hints of risk without a complete source→sink story.
  </confidence_level_definitions>

  <negative_results_documentation>
  - Explicitly record secure vectors in the markdown deliverable: `"username param → bound value at db/auth.sql:14 → slot=SQL-val → safe"`.
  - Negative results prevent re-testing and demonstrate coverage breadth.
  </negative_results_documentation>

  <coverage_requirements>

  - Include background jobs and message-queue consumers if the queue payload is user-influenced.

  </coverage_requirements>
