<screen_methodology>
Process each queue entry independently. For each one, run the xss-specific refutation checks, then render exactly one verdict.

**XSS-specific refutation checks:**
1. **Is the reflection point inside an executable context?** Confirm the named `source` value actually reaches the named `sink_function` and lands in an HTML context where script can execute (HTML body, attribute that breaks out, `javascript:` URL, code-eval sink). If the value is never reflected, or is reflected only inside a context where it cannot execute (e.g. a text node that is fully entity-encoded), the hypothesis is refuted.
2. **Is encoding / CSP / framework escaping in place?** Check the live response for output encoding correct for the `render_context`, and read the response's `Content-Security-Policy`. Context-correct encoding that survives your breakout attempt, or a CSP that demonstrably blocks inline/script execution with no candidate bypass, refutes the hypothesis. (Encoding that is wrong for the actual context — e.g. HTML-entity encoding applied inside a JS string — does NOT refute.)
3. **Benign payload observation:** Inject a minimal, non-malicious payload (e.g. `<script>alert(1)</script>`, an attribute-breakout `"><svg onload=print(1)>`, or a `javascript:print(1)` URL as appropriate to the context) and observe the rendered response/DOM. Script that executes supports passing. Script that is neutralized (encoded, stripped, CSP-blocked) after a fair attempt supports rejection.

**Verdict:**
- `screen_passed` — you independently observed (or plausibly could not rule out) script execution in the claimed context, OR you could neither confirm nor affirmatively refute the finding. The benefit of the doubt goes to passing.
- `screen_rejected` — you hold affirmative evidence the finding is NOT exploitable: the value is not reflected into an executable context, output encoding is correct for the actual render context, or a CSP/framework defense blocked execution with no candidate bypass. Record the concrete reason.

**Conservative cannot-reach rule (MANDATORY):**
If you cannot reach the endpoint or reproduce the conditions to test the finding (auth-gated and you hold no usable session, stored-XSS write path you cannot trigger, rate-limited, endpoint unresponsive), mark it `screen_passed` with reason 'unreachable — cannot refute'. Do not drop a finding you cannot test. False negatives (passing a weak finding) are acceptable; false positives (dropping a real finding) are not.
</screen_methodology>
