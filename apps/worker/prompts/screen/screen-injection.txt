<screen_methodology>
Process each queue entry independently. For each one, run the injection-specific refutation checks, then render exactly one verdict.

**Injection-specific refutation checks:**
1. **Does the reported input actually reach a sink?** Trace the data flow yourself (live probe and/or read-only source inspection via the Task Agent). If the named `source` parameter does not exist on the named endpoint, or the value provably never reaches the `sink_call`, the hypothesis is refuted.
2. **Is the sink actually injectable?** Determine whether a context-correct defense is in place for the declared `slot_type`: a parameterised/bound query, a strict allowlist for identifier/keyword slots, an argv-array (not shell-string) command exec, or an effective escape. A correct, un-bypassed defense at the slot refutes the hypothesis. (A value-context defense on a structural slot is NOT a correct defense — that does NOT refute.)
3. **Benign proof-of-concept:** Send a minimal, non-destructive probe (an error-inducing character, a boolean true/false pair, or a short benign time delay — NEVER a payload that mutates or exfiltrates data). Observe whether it produces an observable, injection-consistent difference (DB error, boolean-divergent responses, measurable delay). No observable difference after a fair attempt, combined with an in-place correct defense, supports rejection. An observable difference supports passing.

**Verdict:**
- `screen_passed` — you independently reproduced injection-consistent behavior, OR you could neither confirm nor affirmatively refute the finding. The benefit of the doubt goes to passing.
- `screen_rejected` — you hold affirmative evidence the finding is NOT exploitable: the input vector does not exist, the value provably never reaches the sink, or a context-correct defense withstood your probes. Record the concrete reason.

**Conservative cannot-reach rule (MANDATORY):**
If you cannot reach the endpoint or reproduce the conditions to test the finding (auth-gated and you hold no usable session, rate-limited, endpoint unresponsive, infrastructure missing), mark it `screen_passed` with reason 'unreachable — cannot refute'. Do not drop a finding you cannot test. False negatives (passing a weak finding) are acceptable; false positives (dropping a real finding) are not.
</screen_methodology>
