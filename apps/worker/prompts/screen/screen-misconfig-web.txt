<screen_methodology>
Process each queue entry independently. For each, run the misconfig-specific refutation checks, then render exactly one verdict.

**Misconfig-specific refutation checks (pick those that fit `vulnerability_type`):**
1. **CORS:** send one preflight with a foreign `Origin`. If the server does NOT reflect that origin (or omits `Allow-Credentials`), the control is safe — refuted. Reflection + credentials supports passing.
2. **CSP / headers:** read the live header. A present CSP without `unsafe-inline`/wildcard for the relevant directive refutes a "missing/weak CSP" claim for that directive.
3. **Open redirect:** request the sink once with a benign off-site target. A server that returns a relative or allow-listed `Location` (ignoring the injected target) refutes it.
4. **Request smuggling:** do NOT fire a desync payload in the screen. Confirm from source/headers whether the proxy and app agree on framing; if the parser demonstrably rejects ambiguous framing, refute. If undecidable, pass.
5. **JWT verify:** one benign probe — does a modified `alg`/claim get rejected? Rejection (signature/claim enforced) refutes; acceptance supports passing.

**Verdict:**
- `screen_passed` — you independently observed (or could not rule out) the control being mis-set, OR you could neither confirm nor affirmatively refute it. Benefit of the doubt goes to passing.
- `screen_rejected` — you hold affirmative evidence the control is safe. Record the concrete reason.

**Conservative cannot-reach rule (MANDATORY):**
If you cannot reach the endpoint or reproduce the conditions (proxy not exercisable, rate-limited, endpoint unresponsive), mark it `screen_passed` with reason 'unreachable — cannot refute'. Do NOT drop a finding you cannot test. False negatives are acceptable; false positives are not.
</screen_methodology>
