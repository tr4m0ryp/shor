<screen_methodology>
Process each queue entry independently. For each one, run the authz-specific refutation checks, then render exactly one verdict.

**Authz-specific refutation checks:**
1. **Does the object/resource actually lack a server-side ownership check?** Using A/B replay, request the resource as an identity that should NOT have access. If the server returns 401/403 or omits the protected data, an ownership/role check is enforced — that refutes the finding. (Use the `minimal_witness` as your starting probe.)
2. **Is the control at the route level or the resource level?** A route-level guard that still lets one authenticated user reach another user's object (no per-object ownership check) is the vulnerability; a guard that is correctly enforced per-object is a working defense. Confirm which one the server actually applies for THIS `endpoint` and `side_effect`.
3. **Cross-identity / modified-object-ID attempt (benign):** Access the resource as a different simulated identity, or with a modified object ID, and check whether the server returns data (or performs the `side_effect`) it should not. Returned cross-identity data supports passing; a server that consistently denies the cross-identity request after a fair attempt supports rejection.

**Verdict:**
- `screen_passed` — you independently observed (or could not rule out) cross-identity access / an unauthorized side effect, OR you could neither confirm nor affirmatively refute the finding. The benefit of the doubt goes to passing.
- `screen_rejected` — you hold affirmative evidence the finding is NOT exploitable: the server enforces a resource-level ownership/role check, or the cross-identity request was consistently denied with the control demonstrably in place. Record the concrete reason.

**Conservative cannot-reach rule (MANDATORY):**
If you cannot reach the endpoint or reproduce the conditions to test the finding (you cannot obtain two distinct identities/sessions, the resource requires state you cannot create, rate-limited, endpoint unresponsive), mark it `screen_passed` with reason 'unreachable — cannot refute'. Do not drop a finding you cannot test. False negatives (passing a weak finding) are acceptable; false positives (dropping a real finding) are not.
</screen_methodology>
