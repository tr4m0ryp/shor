<screen_methodology>
Process each queue entry independently. For each, run the logic-specific refutation checks, then render exactly one verdict.

**Logic-specific refutation checks (pick those that fit `vulnerability_type`):**
1. **Is the invariant actually enforced server-side?** Using the `minimal_witness` as your starting probe against your OWN resource: attempt the forbidden state/transition/bind ONCE. If the server rejects it (4xx, unchanged state, ignored privileged field), the invariant is enforced — that refutes the finding.
2. **Workflow / single-use:** try the step out of order or replay the single-use token once. A server that requires the prior server-verified state, or rejects the replay, refutes it.
3. **Mass assignment:** submit the self-update with ONE privileged field added (`role`/`is_admin`/`price`), then read back. If the field did not change, the bind is allow-listed — refuted.
4. **Race/TOCTOU:** do NOT attempt a real concurrency attack in the screen. Confirm from source whether the check-then-act is transactional; if it demonstrably is (single atomic statement / row lock), refute. If you cannot tell, pass.

**Verdict:**
- `screen_passed` — you independently observed (or could not rule out) the invariant being violable, OR you could neither confirm nor affirmatively refute it. Benefit of the doubt goes to passing.
- `screen_rejected` — you hold affirmative evidence the invariant IS enforced (server rejected the abuse, field not bindable, transaction present). Record the concrete reason.

**Conservative cannot-reach rule (MANDATORY):**
If you cannot reach the endpoint or reproduce the precondition state (cannot create the required state, rate-limited, endpoint unresponsive), mark it `screen_passed` with reason 'unreachable — cannot refute'. Do NOT drop a finding you cannot test. False negatives (passing a weak finding) are acceptable; false positives (dropping a real finding) are not.
</screen_methodology>
