<screen_methodology>
Process each queue entry independently. For each one, run the ssrf-specific refutation checks, then render exactly one verdict.

**SSRF-specific refutation checks:**
1. **Is the URL parameter actually forwarded server-side?** Confirm the `vulnerable_parameter` reaches a server-side HTTP client at `vulnerable_code_location`, rather than being consumed client-side (e.g. used only to build a link in the browser, an `<img>` the browser loads, or a redirect the client follows). A purely client-side parameter is not SSRF — that refutes the finding.
2. **Does the server fetch the supplied URL, or just validate/store it?** Determine whether supplying a URL causes an outbound server request at all. If the server only validates the URL syntactically and stores it without ever fetching it, the hypothesis is refuted. Also check for an effective defense (allowlist of hosts/schemes, internal-range blocking) that withstands a benign probe.
3. **Benign callback probe:** Supply a known-safe external callback URL (a unique `interactsh-client` nonce subdomain, or the scan's own callback URL if one is available) as the parameter value and observe whether the server initiates the request to it. A received callback supports passing; no callback combined with an in-place allowlist/filter supports rejection.

**Verdict:**
- `screen_passed` — you independently observed (or could not rule out) a server-initiated request driven by the parameter, OR you could neither confirm nor affirmatively refute the finding. The benefit of the doubt goes to passing.
- `screen_rejected` — you hold affirmative evidence the finding is NOT exploitable: the parameter is client-side only, the server validates/stores but never fetches the URL, or an effective allowlist/SSRF-filter blocked the benign callback. Record the concrete reason.

**Conservative cannot-reach rule (MANDATORY):**
If you cannot reach the endpoint or reproduce the conditions to test the finding (no callback infrastructure available, auth-gated and you hold no usable session, rate-limited, endpoint unresponsive), mark it `screen_passed` with reason 'unreachable — cannot refute'. Do not drop a finding you cannot test. False negatives (passing a weak finding) are acceptable; false positives (dropping a real finding) are not.
</screen_methodology>
